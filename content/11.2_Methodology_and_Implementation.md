# 11.2 Methodology and Implementation — Cybersecurity Architecture

## Overview

This section details the practical methodology by which the cybersecurity architecture introduced in [[11.1_Preparation_Studies_and_Tools|Section 11.1]] was implemented, integrated, and validated against the physical edge-computing hardware of the [[Smart Home Architecture|Hybrid Edge-Computing AI Smart Home]]. The methodology follows a **layered, dual-discipline structure**: a **Blue Team** track concerned with the deployment of continuous defensive monitoring, and a **Red Team** track concerned with adversarial validation of those defenses through controlled attack simulation. This Red Team / Blue Team cycle is iterative — each Red Team finding fed directly back into a Blue Team configuration revision, producing successive hardening passes over the course of the project.

The chapter is organised into four parts: the integration architecture binding the security layers to the physical hardware; the access control implementation governing all [[Mosquitto]] MQTT communication; the Blue Team deployment methodology; and the Red Team attack simulation methodology used to validate it.

---

## 11.2.1 Multi-Layered Defense Architecture and Hardware Integration

### Conceptual Model

The security architecture was designed using a **concentric layering model**, in which each layer assumes the layer beneath it may eventually be bypassed, and is therefore designed to detect or contain a breach rather than to serve as the sole point of failure. This model directly informed the placement of each control relative to the two physical hardware classes in the system: the [[PC Server]] (the Ubuntu 24.04 edge controller) and the [[ESP32]] sensor/actuator nodes.

The five concentric layers, from outermost to innermost, are:

1. **Network perimeter layer** — [[UFW]] firewall rules and [[Suricata]] network intrusion detection, operating on all traffic entering or leaving the [[PC Server]]'s network interface before it reaches any application.
2. **Transport security layer** — TLS 1.3 encryption on the [[Mosquitto]] broker's port 8883 listener, ensuring that even traffic which passes the perimeter layer cannot be read or altered in transit between the [[ESP32]] nodes and the broker.
3. **Application access control layer** — [[Mosquitto ACLs]] and credential authentication, ensuring that even a client which successfully establishes a TLS session cannot read or write topics outside its authorised scope.
4. **Host integrity layer** — [[Wazuh]] File Integrity Monitoring, rootkit detection, and log-based behavioural rules, operating on the [[PC Server]]'s filesystem and process activity to detect compromise that has already bypassed the outer three layers.
5. **Automated response layer** — [[Fail2ban]] and Wazuh active-response, which convert detections at any of the above layers into immediate, autonomous firewall enforcement without requiring operator intervention.

### Why the ESP32 Nodes Cannot Host Security Agents

A foundational architectural constraint shaping this entire methodology is that the [[ESP32]] microcontrollers — being resource-constrained devices with no general-purpose operating system — cannot run a Wazuh agent, a firewall, or any conventional endpoint security software. This is the central reason the security architecture is **edge-centralised** rather than distributed: all detection and enforcement logic is concentrated on the [[PC Server]], and the ESP32 nodes are treated as **untrusted, monitored endpoints** whose behaviour is observed exclusively through the traffic they generate on the [[Mosquitto]] broker and the network segment.

This has a direct methodological consequence: every security control applied to "the ESP32 nodes" in this architecture is, in practical terms, a control applied to the **broker-side representation** of that node — its MQTT credential, its ACL entry, and the network flow analysis Suricata performs against its traffic pattern. An ESP32 node is never trusted based on physical possession; it is trusted only to the extent that its credentials are valid and its traffic pattern matches expected behaviour.

### Integration Points Between Security Layers and Hardware

The following table maps each defensive layer explicitly onto the hardware component it protects and the mechanism of integration:

| Hardware Component | Defensive Layer | Integration Mechanism |
|---|---|---|
| [[ESP32]] (×4 nodes) | Transport security | TLS 1.3 client connection to broker port 8883, validated against project CA |
| [[ESP32]] (×4 nodes) | Application access control | Per-device Mosquitto credential + topic-scoped ACL entry |
| [[ESP32]] (×4 nodes) | Network perimeter (indirect) | Suricata MQTT protocol parser inspects flows sourced from ESP32 IP range |
| [[PC Server]] | Network perimeter | UFW default-deny inbound policy; Suricata AF_PACKET capture on LAN interface |
| [[PC Server]] | Host integrity | Wazuh Agent — FIM on `/etc/mosquitto`, `/opt/smarthome`; rootcheck; log monitoring |
| [[PC Server]] | Automated response | Fail2ban jails bound to UFW `firewall-drop`; Wazuh active-response |
| [[Mosquitto]] broker process | Application access control | `allow_anonymous false`; `password_file`; `acl_file` |
| [[AI Decision Engine]] | Application access control | Dedicated `ai_engine` MQTT credential with broad but explicitly scoped ACL |

This table is the practical implementation of the architecture diagram presented in [[11.1_Preparation_Studies_and_Tools#Summary of Tool Integration|Section 11.1's tool integration summary]]; where that diagram showed data flow, this table shows the corresponding **trust boundary** at each hop.

### The Edge Server as a Single Point of Convergence

A deliberate design tension exists in this architecture: the [[PC Server]] is simultaneously the most heavily defended node (hosting the entire Blue Team stack) and the highest-value target (since it is the sole aggregation point for sensor data, the sole issuer of actuation commands, and the sole node with cloud connectivity to [[ThingsBoard]]). This was explicitly identified during the architecture phase as an unavoidable consequence of the single-host constraint of the graduation project scope, and is addressed methodologically by ensuring no single defensive layer is solely responsible for protecting the server — a finding that directly motivated the Red Team exercises described in [[#11.2.4 Red Team Operations — Attack Simulation Methodology]].

## 11.2.2 Access Control Implementation — Mosquitto ACLs

### Design Principle: Least Privilege per Device Identity

The implementation of [[Mosquitto ACLs]] follows the principle of least privilege applied at the granularity of individual device identity. Rather than a shared credential or a permissive topic structure (e.g., a single `home/#` wildcard grant to all devices), each [[ESP32]] node and each software service was issued a unique Mosquitto credential and a correspondingly narrow ACL entry restricting it to only the topic subtree it legitimately requires.

This design directly mitigates the **lateral movement** risk inherent in shared-credential IoT deployments: in a flat-ACL system, the compromise of a single low-value sensor node (such as a door/window reed switch) would grant an attacker read/write access to every other device's data and control channel. Under the implemented per-device ACL scheme, the same compromise is contained to that single device's topic subtree.

### Topic Namespace Design

The MQTT topic namespace was structured hierarchically to support per-device ACL scoping:

```
home/
├── sensors/
│   ├── esp32_1/#         (temperature, humidity)
│   ├── esp32_2/#         (motion, PIR)
│   ├── esp32_3/#         (door/window reed switch)
│   └── esp32_4/#         (power meter, PZEM-004T)
├── actuators/
│   ├── esp32_1/#
│   ├── esp32_2/#
│   ├── esp32_3/#
│   └── esp32_4/#
├── ai/
│   ├── decisions
│   └── status
└── rpc/#                  (ThingsBoard cloud command channel)
```

This namespace separates **telemetry** (`sensors/`) from **command** (`actuators/`) at the top level, which allows the ACL file to express the asymmetric trust relationship each device has with the broker: a sensor node should generally be permitted to *publish* telemetry but should only be permitted to *subscribe* to commands addressed specifically to it, never to other devices' commands.

### Implemented ACL Configuration

The production ACL file, located at `/etc/mosquitto/acl`, implements the following access matrix:

```conf
# ── ESP32 Sensor 1 (Temperature/Humidity) ───────────────────────
user esp32_sensor_1
topic write home/sensors/esp32_1/#
topic read  home/actuators/esp32_1/#

# ── ESP32 Sensor 2 (Motion/PIR) ──────────────────────────────────
user esp32_sensor_2
topic write home/sensors/esp32_2/#
topic read  home/actuators/esp32_2/#

# ── ESP32 Sensor 3 (Door/Window) ─────────────────────────────────
user esp32_sensor_3
topic write home/sensors/esp32_3/#
topic read  home/actuators/esp32_3/#

# ── ESP32 Sensor 4 (Power Meter) ─────────────────────────────────
user esp32_sensor_4
topic write home/sensors/esp32_4/#
topic read  home/actuators/esp32_4/#

# ── AI Decision Engine ────────────────────────────────────────────
user ai_engine
topic read  home/sensors/#
topic write home/actuators/#
topic write home/ai/decisions
topic write home/ai/status

# ── ThingsBoard Cloud Gateway ──────────────────────────────────────
user tb_gateway
topic read  home/#
topic write home/rpc/#
```

### Rationale for Each Access Grant

The `esp32_sensor_N` identities are granted **write-only** access to their own sensor subtree. This reflects the unidirectional nature of telemetry: a temperature sensor has no legitimate reason to read its own published data back. The corresponding **read-only** grant on the matching actuator subtree exists only for nodes that also host an actuator (for example, a combined sensor/relay module); for pure-sensor nodes, this grant remains unused but is retained for architectural symmetry and future hardware revisions.

The `ai_engine` identity is the only credential granted a topic-level wildcard (`home/sensors/#`) for read access, because the [[AI Decision Engine]] requires simultaneous visibility into all sensor streams to perform cross-sensor inference (for example, correlating a motion event with a temperature reading to decide on HVAC actuation). This represents the single broadest privilege grant in the system and was therefore the primary focus of the FIM monitoring on `/opt/smarthome` described in Section 11.1 — any unauthorised modification to the AI engine's code is treated as a critical-severity event precisely because it inherits this broad MQTT privilege.

The `tb_gateway` identity is granted read-only access across the entire namespace, reflecting its role as a passive telemetry forwarder to [[ThingsBoard]], with write access scoped exclusively to the `home/rpc/#` subtree through which the cloud platform issues remote commands back down to the edge.

### Enforcement Verification Methodology

ACL enforcement was verified through manual penetration testing using the `mosquitto_pub` and `mosquitto_sub` CLI clients, in which each device credential was deliberately used to attempt access outside its authorised topic scope. A representative test case is shown below, in which `esp32_sensor_1`'s credential is used to attempt a write to `esp32_sensor_2`'s topic:

```bash
mosquitto_pub -h localhost -p 8883 \
  --cafile /etc/mosquitto/certs/ca.crt \
  -u esp32_sensor_1 -P [REDACTED] \
  -t "home/sensors/esp32_2/dht22" \
  -m '{"temp": 99}'
# Result: Connection accepted, publish rejected by broker (ACL denied)
```

Each ACL-denied event of this nature generates a Mosquitto log entry matched by the custom Wazuh decoder described in Section 11.1, triggering rule `100003` at severity level 8. This confirms an end-to-end enforcement chain: **ACL violation → broker rejection → log entry → Wazuh detection → SIEM-visible alert**, validating that the access control layer is not merely preventative but also observable.

## 11.2.3 Blue Team Deployment Methodology — Continuous Defensive Monitoring

### Deployment Philosophy

The Blue Team deployment methodology was executed in five sequential phases, each building upon the verified functionality of the previous phase. This phased approach was adopted specifically because [[Wazuh]] and [[Suricata]] depend on correctly functioning lower-layer services ([[Mosquitto]], UFW) to have meaningful log data to analyse; deploying the detection layer before the data-producing layer would have resulted in an untestable monitoring configuration.

### Phase 1 — Baseline Hardening Prior to Monitoring Deployment

Before any detection tooling was installed, the [[PC Server]] was brought to the hardened baseline state described in Section 11.1: [[UFW]] default-deny policy, SSH hardening (key-only authentication, non-default port), and kernel `sysctl` parameters. This ordering ensures that the subsequent monitoring deployment observes a system already conforming to its intended security posture, rather than capturing a noisy transition period during which insecure defaults were still in effect.

### Phase 2 — Wazuh Manager, Indexer, and Dashboard Deployment

The Wazuh Manager, Indexer, and Dashboard components were deployed in strict dependency order: Indexer first (as the data store the Manager depends upon), Manager second, Dashboard last (as the presentation layer dependent on the Manager's API). Each component's service state and listening port were explicitly verified via `systemctl is-active` and `ss -tlnp` before proceeding to the next, rather than assuming successful installation from package manager exit codes alone. This verification-gated approach was adopted directly in response to early deployment failures in which a partially-removed package (`dpkg` state `rc`) silently left the Wazuh Manager service absent from `systemd` despite the Indexer and Dashboard reporting healthy status — a failure mode invisible without explicit per-component verification.

### Phase 3 — Wazuh Agent Registration and FIM Activation

The Wazuh Agent was installed on the [[PC Server]] itself (registering it as a monitored endpoint of its own SIEM, agent ID `001`) and configured to point to the local Manager instance at `127.0.0.1`. Agent registration was verified using `agent_control -l` to confirm an `Active` status before proceeding, since an agent in `Never connected` or `Disconnected` state produces no log forwarding despite appearing nominally "installed."

Following successful registration, File Integrity Monitoring was activated with `realtime="yes"` scanning on the directory set defined in Section 11.1, and validated by deliberately modifying a monitored file (`/etc/mosquitto/mosquitto.conf`) and confirming the corresponding FIM alert appeared in the Wazuh Dashboard within the expected real-time detection window.

### Phase 4 — Suricata Network Monitoring Activation

[[Suricata]] was deployed with the AF_PACKET capture mode bound to the [[PC Server]]'s active LAN interface, identified dynamically via `ip route` rather than hardcoded, since interface naming varies across deployment environments (`eth0`, `ens33`, `enp0s3`, etc., depending on the underlying virtualisation or NIC driver). The community ruleset (ET/Open) was installed via `suricata-update` to provide baseline signature coverage, and the project-specific MQTT and SSH rule set described in Section 11.1 was layered on top.

Suricata's operational correctness was validated using the built-in configuration test mode (`suricata -T -c suricata.yaml -v`) prior to enabling the systemd service, and live packet capture was confirmed by tailing `/var/log/suricata/eve.json` while generating known traffic (an `mosquitto_pub` TLS handshake) and confirming the corresponding `tls` and `flow` event types appeared in the EVE log.

### Phase 5 — Cross-Tool Alert Correlation Pipeline

The final Blue Team deployment phase established the alert correlation pipeline connecting Wazuh and Suricata output into [[Splunk]]. Suricata's `eve.json` was configured as a Wazuh `<localfile>` monitor target, meaning Suricata alerts are not only visible in the Suricata-native log but also re-processed through the Wazuh ruleset (rule `100009`, mapped to MITRE ATT&CK T1499). Wazuh's own alert stream was simultaneously forwarded to Splunk via the HTTP Event Collector integration described in Section 11.1, establishing a single point of query (Splunk) capable of correlating host-based and network-based detections from a unified search interface.

### Continuous Monitoring Operating Procedure

Following deployment, the Blue Team operating procedure consists of the following recurring activities, each mapped to a specific dashboard or query:

| Activity | Tool | Method |
|---|---|---|
| Review new high/critical severity alerts | Wazuh Dashboard | Filter `rule.level >= 12` |
| Review MITRE ATT&CK technique coverage | Wazuh Dashboard | MITRE ATT&CK module |
| Review network-layer alerts | Suricata / Splunk | `index=suricata sourcetype=suricata_eve event_type=alert` |
| Review currently banned IPs | Fail2ban | `fail2ban-client status <jail>` |
| Cross-correlate host + network alerts | Splunk | Join on `community_id` field |
| Verify FIM baseline integrity | Wazuh Dashboard | File Integrity Monitoring module |

## 11.2.4 Red Team Operations — Attack Simulation Methodology

### Purpose and Scope

The Red Team methodology was designed to empirically validate the Blue Team defenses described above by simulating the specific attack techniques the detection rules were authored to catch, rather than performing untargeted or exploratory penetration testing. Each Red Team exercise was scoped to a single MITRE ATT&CK technique already mapped to a custom Wazuh rule or Suricata signature, such that a successful exercise produces a direct, falsifiable test of whether the corresponding Blue Team control fires as designed. All Red Team activity was conducted exclusively against the isolated project test network, using only project-owned hardware, with no external network targets at any point.

### Exercise 1 — MQTT Credential Brute Force (T1110.001)

**Objective:** Validate that Wazuh rule `100002` and the Fail2ban `mosquitto` jail correctly detect and respond to repeated authentication failures against the broker's TLS listener.

**Method:** A scripted loop issued repeated `mosquitto_pub` connection attempts against port 8883 using a deliberately invalid credential, at a rate exceeding five attempts within a two-minute window — the threshold configured in the `100002` rule's `frequency`/`timeframe` parameters.

```bash
for i in {1..6}; do
  mosquitto_pub -h 192.168.1.100 -p 8883 \
    --cafile /etc/mosquitto/certs/ca.crt \
    -u attacker_test -P invalid_credential \
    -t "home/test" -m "probe" 2>&1
  sleep 5
done
```

**Expected Defensive Response:** A Mosquitto log entry per failed attempt; Wazuh rule `100001` firing on each individual failure; rule `100002` firing once the frequency threshold is crossed; Fail2ban's `mosquitto` jail independently detecting the same pattern via its own filter and issuing a `firewall-drop` ban through UFW.

**Validation Method:** Post-exercise inspection of `fail2ban-client status mosquitto` confirmed the test source IP appeared in the banned IP list, and the Wazuh Dashboard's Security Events module confirmed both rule `100001` and `100002` had fired with timestamps consistent with the attack window.

### Exercise 2 — ACL Boundary Violation (T1078)

**Objective:** Validate that the access control implementation described in [[#11.2.2 Access Control Implementation — Mosquitto ACLs]] correctly rejects cross-device topic access even when presented with valid authentication credentials, and that this rejection is independently observable in the SIEM.

**Method:** Using the legitimately issued and valid credential for `esp32_sensor_1`, an attempt was made to publish to the topic subtree reserved for `esp32_sensor_2`, simulating a scenario in which a single device's credential had been extracted (e.g., via physical access to flash memory) but the attacker had not gained broker-level administrative access.

**Expected Defensive Response:** Broker-level rejection of the publish operation despite valid TLS and authentication; a corresponding `ACL check denied` log entry; Wazuh rule `100003` firing at severity level 8.

**Validation Method:** Confirmed via direct inspection of `/var/log/mosquitto/mosquitto.log` for the ACL denial entry and cross-confirmed against the Wazuh Dashboard alert corresponding to rule `100003`. This exercise specifically validates that authentication and authorisation are enforced as independent controls, since a system that conflates the two would have permitted this cross-device write.

### Exercise 3 — TLS Certificate Tampering Simulation (T1557)

**Objective:** Validate that File Integrity Monitoring on the certificate directory (`/etc/mosquitto/certs`) is sufficiently sensitive to detect the preparatory phase of a Man-in-the-Middle attack, in which an adversary would need to introduce or replace a TLS certificate before being able to intercept broker traffic.

**Method:** A non-production test certificate was deliberately written into `/etc/mosquitto/certs/` to simulate an unauthorised certificate substitution, performed during a maintenance window to avoid disrupting live broker operation.

**Expected Defensive Response:** Immediate (realtime, `inotify`-driven) Wazuh FIM alert, given the directory's `realtime="yes"` configuration, escalating to rule `100006` at severity level 14 (Critical) — the highest severity level defined in the custom rule set, reflecting the gravity of credential infrastructure tampering.

**Validation Method:** Time-to-detection was measured from file write timestamp to Wazuh Dashboard alert appearance, confirming sub-minute detection latency consistent with the realtime FIM configuration rather than the 300-second polling interval used for non-realtime monitored directories.

### Exercise 4 — Network Reconnaissance and Port Scanning (T1595.002 / T1046)

**Objective:** Validate that [[Suricata]]'s network-layer monitoring detects reconnaissance activity against the broker's TLS port prior to any application-layer authentication attempt, providing an earlier detection point than the Mosquitto-log-based rules in Exercises 1 and 2.

**Method:** A standard TCP port scan was executed against the [[PC Server]]'s LAN-facing interface, targeting the known service ports (8883, 2222, 443) as well as a range of unused ports to additionally validate that UFW's default-deny policy generates appropriately logged drops for the unused range.

**Expected Defensive Response:** Suricata alert generation for the scanning traffic pattern, matched against both the community ruleset and the project-specific custom rule targeting port scan behaviour against 8883; corresponding UFW log entries for connection attempts against ports with no listening service; Fail2ban's `suricata` jail (`maxretry = 1`) issuing an immediate ban upon the first Suricata alert.

**Validation Method:** Confirmed via `eve.json` inspection for `alert` event types correlated by `community_id`, cross-referenced against `/var/log/ufw.log` drop entries for the unused port range, and confirmed in `fail2ban-client status suricata` for the resulting ban.

### Exercise 5 — Configuration Tampering Detection (T1565.001)

**Objective:** Validate detection of unauthorised modification to the Mosquitto broker's own configuration file, simulating a scenario in which an attacker who has gained some level of host access attempts to weaken the broker's security posture directly (for example, by re-enabling anonymous access).

**Method:** A controlled, reversible modification was made to `/etc/mosquitto/mosquitto.conf` during a maintenance window, altering a non-critical comment line to avoid disrupting the live configuration while still triggering the file-level FIM hash change.

**Expected Defensive Response:** Wazuh FIM detection of the file modification; rule `100005` firing at severity level 12, distinct from and lower than the level-14 certificate tampering rule, reflecting the methodological decision to treat credential/certificate compromise as categorically more severe than general configuration drift.

**Validation Method:** Confirmed via Wazuh Dashboard FIM module showing the before/after file hash and the corresponding alert entry.

### Iterative Feedback Into Blue Team Configuration

Each of the five exercises above was conducted at least twice: an initial run against the as-deployed configuration, and a confirmation run following any configuration adjustment necessitated by gaps identified in the initial run. This iterative loop is the methodological core of the Red Team / Blue Team relationship in this project — the Red Team exercises were not a one-time validation step performed after defensive deployment was declared complete, but an integrated part of the deployment process itself, with each exercise result treated as a requirement against which the corresponding Wazuh rule, Suricata signature, or Fail2ban jail was tuned until the expected detection behaviour was empirically confirmed.

| Exercise | MITRE ATT&CK | Primary Detection Control | Secondary Detection Control | Response Control |
|---|---|---|---|---|
| MQTT credential brute force | T1110.001 | Wazuh rule 100002 | — | Fail2ban `mosquitto` jail |
