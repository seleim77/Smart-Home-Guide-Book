# Splunk Enterprise — SIEM Integration and Correlation

## 1. Introduction and Role in the Security Stack

[[Splunk]] Enterprise occupies the **top-level aggregation and correlation layer** of the smart home cybersecurity architecture. Where [[Wazuh]] provides host-based detection and [[Suricata]] provides network-based detection, Splunk provides neither detection nor enforcement independently — instead, it is the platform through which the outputs of all other security controls are unified into a single, queryable corpus. Its primary operational value is enabling **cross-tool correlation**: the identification of attack patterns that span the detection boundary between the network layer (Suricata alerts) and the host layer (Wazuh alerts), which neither tool could produce independently.

Splunk Enterprise 9.2.1 was installed on the [[PC Server]] from the official `.deb` package. The free tier (500 MB/day ingestion limit) is sufficient for all data volumes generated by this deployment, which were measured during normal operation at approximately 15–40 MB/day depending on sensor activity and security event frequency.

## 2. Deployment Configuration

### 2.1 Service and Interface Binding

The `Splunkd` systemd service was enabled for boot-start and the web interface bound exclusively to the loopback interface to prevent LAN exposure:

```ini
# /opt/splunk/etc/system/local/web.conf
[settings]
httpport = 8000
enableSplunkWebSSL = false
server.socket_host = 127.0.0.1
```

UFW enforcement:

```bash
sudo ufw allow from 127.0.0.1 to any port 8000 comment 'Splunk UI local only'
sudo ufw allow from 127.0.0.1 to any port 8088 comment 'Splunk HEC local only'
sudo ufw allow from 127.0.0.1 to any port 8089 comment 'Splunk API local only'
sudo ufw deny 8000
sudo ufw deny 8088
sudo ufw deny 8089
```

This ensures the Splunk administrative interface, HEC endpoint, and REST API are accessible only from a local session on the server — consistent with the access restriction policy applied to all other internal services (Wazuh Dashboard on port 443, Wazuh Indexer on port 9200).

### 2.2 Port Allocation

| Port | Service | Binding | Purpose |
|---|---|---|---|
| `8000` | Splunk Web UI | `127.0.0.1` | Browser-based search and dashboard interface |
| `8088` | HTTP Event Collector | `127.0.0.1` | Real-time event ingestion from Wazuh Manager |
| `8089` | Splunk REST API | `127.0.0.1` | CLI management, search dispatch, monitor configuration |
| `9997` | Universal Forwarder (not used) | n/a | Not deployed — all data sources are local |

## 3. Index Architecture

### 3.1 Design Rationale

Data partitioning across multiple indexes rather than a single catch-all index provides three operational benefits in this deployment:

1. **Search performance** — Splunk searches are bounded to specified indexes, eliminating the overhead of scanning event types irrelevant to a given query.
2. **Retention policy independence** — each index can be assigned a different `maxTotalDataSizeMB` limit and `frozenTimePeriodInSecs` retention period, allowing high-volume low-priority data (Mosquitto connection logs) to age out faster than high-value alert data (Wazuh alerts).
3. **Access control granularity** — Splunk role-based access control operates at the index level, enabling future multi-user deployments where, for example, a network operator has access to `suricata` but not `wazuh_alerts`.

### 3.2 Index Definitions

```bash
# mqtt_logs — Mosquitto broker events
sudo /opt/splunk/bin/splunk add index mqtt_logs \
  -maxTotalDataSizeMB 2048 \
  -auth admin:SmartHome@Splunk2025!

# wazuh_alerts — Wazuh Manager processed alerts (JSON)
sudo /opt/splunk/bin/splunk add index wazuh_alerts \
  -maxTotalDataSizeMB 5120 \
  -auth admin:SmartHome@Splunk2025!

# suricata — Suricata EVE JSON events
sudo /opt/splunk/bin/splunk add index suricata \
  -maxTotalDataSizeMB 2048 \
  -auth admin:SmartHome@Splunk2025!

# smarthome — Auth, UFW, Fail2ban, Syslog
sudo /opt/splunk/bin/splunk add index smarthome \
  -maxTotalDataSizeMB 5120 \
  -auth admin:SmartHome@Splunk2025!
```

| Index | Primary Sources | Max Size | Retention Priority |
|---|---|---|---|
| `mqtt_logs` | `/var/log/mosquitto/mosquitto.log` | 2 GB | Medium |
| `wazuh_alerts` | `/var/ossec/logs/alerts/alerts.json`, HEC | 5 GB | High |
| `suricata` | `/var/log/suricata/eve.json` | 2 GB | High |
| `smarthome` | `/var/log/auth.log`, `/var/log/ufw.log`, `/var/log/fail2ban.log`, `/var/log/syslog` | 5 GB | Medium |

## 4. Data Ingestion Pipeline

### 4.1 Universal Monitor Inputs

Splunk's universal monitor (`inputs.conf`) was configured to tail all local log files. The monitor inputs are defined through CLI addition rather than direct file editing to ensure correct service registration:

```bash
# Mosquitto MQTT broker log
sudo /opt/splunk/bin/splunk add monitor \
  /var/log/mosquitto/mosquitto.log \
  -index mqtt_logs \
  -sourcetype mosquitto \
  -auth admin:SmartHome@Splunk2025!

# Wazuh JSON alerts
sudo /opt/splunk/bin/splunk add monitor \
  /var/ossec/logs/alerts/alerts.json \
  -index wazuh_alerts \
  -sourcetype wazuh_json \
  -auth admin:SmartHome@Splunk2025!

# Suricata EVE JSON
sudo /opt/splunk/bin/splunk add monitor \
  /var/log/suricata/eve.json \
  -index suricata \
  -sourcetype suricata_eve \
  -auth admin:SmartHome@Splunk2025!

# Fail2ban
sudo /opt/splunk/bin/splunk add monitor \
  /var/log/fail2ban.log \
  -index smarthome \
  -sourcetype fail2ban \
  -auth admin:SmartHome@Splunk2025!

# SSH authentication
sudo /opt/splunk/bin/splunk add monitor \
  /var/log/auth.log \
  -index smarthome \
  -sourcetype linux_secure \
  -auth admin:SmartHome@Splunk2025!

# UFW firewall drops
sudo /opt/splunk/bin/splunk add monitor \
  /var/log/ufw.log \
  -index smarthome \
  -sourcetype ufw \
  -auth admin:SmartHome@Splunk2025!

# General system log
sudo /opt/splunk/bin/splunk add monitor \
  /var/log/syslog \
  -index smarthome \
  -sourcetype syslog \
  -auth admin:SmartHome@Splunk2025!
```

### 4.2 Sourcetype Parsing Configuration — `props.conf`

The `props.conf` file instructs Splunk how to parse timestamps and structure fields for each sourcetype:

```ini
# /opt/splunk/etc/system/local/props.conf

[mosquitto]
# Mosquitto timestamp: 2024-01-15T14:32:07
TIME_FORMAT = %Y-%m-%dT%H:%M:%S
TIME_PREFIX = ^
MAX_TIMESTAMP_LOOKAHEAD = 20
SHOULD_LINEMERGE = false
category = Smart Home / IoT

[wazuh_json]
# Wazuh alert timestamp: "2024-01-15T14:32:07.123+0200"
KV_MODE = json
TIME_FORMAT = %Y-%m-%dT%H:%M:%S.%f%z
TIME_PREFIX = \"timestamp\":\"
SHOULD_LINEMERGE = false
category = Security / HIDS

[suricata_eve]
# Suricata EVE timestamp: "2024-01-15T14:32:07.456789+0200"
KV_MODE = json
TIME_FORMAT = %Y-%m-%dT%H:%M:%S.%f%z
TIME_PREFIX = \"timestamp\":\"
SHOULD_LINEMERGE = false
category = Security / NIDS

[fail2ban]
# Fail2ban timestamp: 2024-01-15 14:32:07,123
TIME_FORMAT = %Y-%m-%d %H:%M:%S,%3N
TIME_PREFIX = ^
SHOULD_LINEMERGE = false
category = Security / Response

[ufw]
# UFW timestamp uses syslog header — Splunk extracts automatically
SHOULD_LINEMERGE = false
category = Security / Firewall

[linux_secure]
SHOULD_LINEMERGE = false
category = Security / Authentication
```

`KV_MODE = json` on the `wazuh_json` and `suricata_eve` sourcetypes instructs Splunk to automatically extract all JSON fields as indexed fields without requiring explicit `EXTRACT` or `FIELDALIAS` transforms. This means fields like `rule.id`, `rule.level`, `alert.signature`, `community_id`, and `src_ip` are immediately available as filterable fields in every search.

## 5. HTTP Event Collector — Real-Time Wazuh Integration

### 5.1 Architecture

The universal monitor inputs described above provide **file-based, near-real-time** ingestion with a polling latency of approximately 1–5 seconds. For the `wazuh_alerts` index, a second, parallel ingestion path was established using Splunk's **HTTP Event Collector (HEC)**, which provides genuine real-time push ingestion with sub-second latency from the Wazuh Manager.

The HEC integration means that when a Wazuh rule fires (for example, rule 100002 on MQTT brute force), the Wazuh Manager immediately HTTP POSTs the alert JSON to Splunk's HEC endpoint at `http://127.0.0.1:8088/services/collector`, before the alert has even been written to disk. This is critical for time-sensitive Blue Team correlation queries that should reflect the current alert state rather than a state that is seconds stale.

### 5.2 HEC Token Creation

```bash
# Enable the HEC service globally
sudo /opt/splunk/bin/splunk http-event-collector enable \
  -uri https://localhost:8089 \
  -auth admin:SmartHome@Splunk2025!

# Create a dedicated token for Wazuh
sudo /opt/splunk/bin/splunk http-event-collector create wazuh-hec-token \
  -index wazuh_alerts \
  -sourcetype wazuh_json \
  -uri https://localhost:8089 \
  -auth admin:SmartHome@Splunk2025!

# Retrieve the generated token value
sudo /opt/splunk/bin/splunk http-event-collector list \
  -uri https://localhost:8089 \
  -auth admin:SmartHome@Splunk2025! | grep -A2 "wazuh-hec-token"
```

### 5.3 Wazuh Manager Integration Configuration

The `<integration>` block in `/var/ossec/etc/ossec.conf` configures the Manager to push alerts to the HEC endpoint:

```xml
<!-- Splunk HEC integration — real-time alert forwarding -->
<integration>
  <name>splunk</name>
  <!-- HEC endpoint — loopback only, consistent with port restriction policy -->
  <hook_url>http://127.0.0.1:8088/services/collector</hook_url>
  <!-- Token generated in Step 5.2 above -->
  <api_key>YOUR_GENERATED_HEC_TOKEN</api_key>
  <!-- Forward all alerts at severity level 3 and above.
       Level 3 is the minimum meaningful Wazuh severity (informational).
       Levels 1-2 are debug/system messages not relevant to Blue Team. -->
  <alert_level>3</alert_level>
  <alert_format>json</alert_format>
</integration>
```

Following configuration, the Wazuh Manager service was restarted to activate the integration:

```bash
sudo systemctl restart wazuh-manager
```

### 5.4 Ingestion Pipeline Diagram

```
Wazuh Manager
    │
    ├── (1) Writes to disk: /var/ossec/logs/alerts/alerts.json
    │        └── Splunk universal monitor tails file (~1-5s latency)
    │                └── Indexed to: wazuh_alerts
    │
    └── (2) HTTP POST to HEC: http://127.0.0.1:8088/services/collector
             └── Splunk indexes immediately (sub-second latency)
                     └── Indexed to: wazuh_alerts (same index, both paths)

Suricata
    │
    └── Writes to disk: /var/log/suricata/eve.json
             └── Splunk universal monitor tails file (~1-5s latency)
                     └── Indexed to: suricata
```

## 6. SPL Query Library — Blue Team Operations

The following Search Processing Language (SPL) queries were used during Blue Team monitoring and Red Team exercise validation. Each query is annotated with its operational purpose and the corresponding Red Team exercise it was used to validate, as referenced in [[11.2_Methodology_and_Implementation#11.2.4 Red Team Operations — Attack Simulation Methodology|Section 11.2.4]].

### 6.1 Alert Volume and Severity Overview

```spl
index=wazuh_alerts
| eval severity=case(
    rule.level>=14, "Critical",
    rule.level>=10, "High",
    rule.level>=7,  "Medium",
    rule.level>=4,  "Low",
    true(),         "Informational"
  )
| timechart span=1h count by severity
```

**Purpose:** Real-time alert volume trend by severity. Used to identify spikes in activity indicating active attack campaigns or false-positive storms following rule changes.

---

### 6.2 MQTT Brute Force Detection Dashboard Query

```spl
index=wazuh_alerts rule.id IN ("100001","100002")
| eval attack_phase=if(rule.id=="100001","Individual Failure","Brute Force Confirmed")
| stats
    count AS total_events,
    min(_time) AS first_seen,
    max(_time) AS last_seen,
    values(rule.description) AS rule_descriptions
    by agent.ip, attack_phase
| eval duration_seconds = last_seen - first_seen
| eval first_seen=strftime(first_seen,"%Y-%m-%d %H:%M:%S")
| eval last_seen=strftime(last_seen,"%Y-%m-%d %H:%M:%S")
| sort -total_events
```

**Purpose:** Identifies source IPs engaged in MQTT credential brute-force activity, showing both the individual failure count (rule 100001) and confirmed brute-force confirmations (rule 100002). Used to validate Red Team Exercise 1 outcomes.

### 6.3 Cross-Tool Correlation — Suricata Network Alert Joined with Wazuh Host Alert

```spl
index=suricata event_type=alert
| rename community_id AS join_key, src_ip AS attacker_ip
| table _time, attacker_ip, join_key, alert.signature, alert.severity
| join type=inner join_key [
    search index=wazuh_alerts
    | rename network.community_id AS join_key
    | table join_key, rule.id, rule.level, rule.description, agent.hostname
  ]
| eval time_str=strftime(_time,"%Y-%m-%d %H:%M:%S")
| table time_str, attacker_ip, alert.signature, rule.id,
         rule.level, rule.description, agent.hostname, join_key
| sort -rule.level
```

**Purpose:** The primary cross-tool correlation query. Joins Suricata network-layer alerts with Wazuh host-layer alerts on the `community_id` field (see [[Suricata#3.3 The community-id Field — Cross-Tool Correlation Key|Suricata Section 3.3]]). A result row in this query represents an attack that was simultaneously detected at the network layer by Suricata and at the host layer by Wazuh — the highest-confidence alert class in the deployment. Used to validate Red Team Exercise 4 (port scan → brute force chain).

---

### 6.4 ACL Violation Trend — Compromised Device Indicator

```spl
index=wazuh_alerts rule.id="100003"
| spath input=_raw output=client_id path="data.extra_data"
| spath input=_raw output=topic_attempted path="data.url"
| stats
    count AS violation_count,
    dc(topic_attempted) AS unique_topics_attempted,
    values(topic_attempted) AS topics,
    earliest(_time) AS first_seen,
    latest(_time) AS last_seen
    by client_id
| eval first_seen=strftime(first_seen,"%Y-%m-%d %H:%M:%S")
| eval last_seen=strftime(last_seen,"%Y-%m-%d %H:%M:%S")
| where violation_count > 0
| sort -violation_count
```

**Purpose:** Groups ACL violations by the offending client ID. A client ID appearing here represents a device (or a credential) that is attempting to access topics outside its authorised scope. A high `unique_topics_attempted` value for a single client suggests systematic topic enumeration — an attacker with a stolen credential probing the topic namespace. Used to validate Red Team Exercise 2.

---

### 6.5 TLS Certificate FIM Alert Timeline

```spl
index=wazuh_alerts rule.id IN ("100005","100006")
| eval alert_class=case(
    rule.id=="100006", "CRITICAL — Certificate Directory Modified",
    rule.id=="100005", "HIGH — Broker Configuration Modified"
  )
| spath input=_raw output=modified_file path="syscheck.path"
| spath input=_raw output=change_type  path="syscheck.event"
| spath input=_raw output=sha256_before path="syscheck.sha256_before"
| spath input=_raw output=sha256_after  path="syscheck.sha256_after"
| table _time, alert_class, modified_file, change_type,
         sha256_before, sha256_after, agent.hostname
| sort -_time
```

**Purpose:** Displays a timeline of FIM alerts for the MQTT configuration and certificate directories, including the before/after SHA-256 hashes. The hash delta confirms that the file content actually changed (not just the timestamp or access time). Used to validate Red Team Exercise 3 and 5.

### 6.6 Fail2ban Ban Events Correlated with Triggering Alerts

```spl
index=smarthome sourcetype=fail2ban "Ban"
| rex field=_raw "(?i)Ban\s+(?P<banned_ip>\d+\.\d+\.\d+\.\d+)"
| rex field=_raw "\[(?P<jail_name>[^\]]+)\]"
| eval ban_time=_time
| join type=left banned_ip [
    search index=wazuh_alerts (rule.id="100002" OR rule.id="5712")
    | rename srcip AS banned_ip
    | stats
        count AS alert_count,
        min(_time) AS first_alert_time,
        values(rule.description) AS triggering_rules
        by banned_ip
  ]
| eval detection_to_ban_seconds = ban_time - first_alert_time
| eval ban_time=strftime(ban_time,"%Y-%m-%d %H:%M:%S")
| table ban_time, banned_ip, jail_name, alert_count,
         triggering_rules, detection_to_ban_seconds
| sort -ban_time
```

**Purpose:** Correlates Fail2ban IP ban events with the Wazuh alerts that preceded them, computing the `detection_to_ban_seconds` metric — the elapsed time between the first Wazuh alert for an attacker IP and the Fail2ban enforcement action. This metric is reported in [[11.3_Results|Section 11.3]] as a key performance indicator for the automated response layer.

### 6.7 Smart Home Attack Chain — Full Kill Chain View

```spl
(index=suricata event_type=alert) OR
(index=wazuh_alerts rule.id IN ("100001","100002","100003","100007","100008")) OR
(index=smarthome sourcetype=fail2ban Ban)
| eval source_tool=case(
    index="suricata",     "Suricata (Network)",
    index="wazuh_alerts", "Wazuh (Host)",
    index="smarthome",    "Fail2ban (Response)"
  )
| eval event_description=coalesce(
    'alert.signature',
    'rule.description',
    "IP Banned"
  )
| eval attacker_ip=coalesce(src_ip, srcip, banned_ip)
| where isnotnull(attacker_ip)
| stats
    list(source_tool)      AS detection_chain,
    list(event_description) AS event_chain,
    count                   AS total_events,
    min(_time)              AS attack_start,
    max(_time)              AS last_event
    by attacker_ip
| eval attack_start=strftime(attack_start,"%Y-%m-%d %H:%M:%S")
| eval last_event=strftime(last_event,"%Y-%m-%d %H:%M:%S")
| eval duration_seconds = strptime(last_event,"%Y-%m-%d %H:%M:%S")
                          - strptime(attack_start,"%Y-%m-%d %H:%M:%S")
| sort -total_events
```

**Purpose:** The comprehensive attack chain view. For each attacker IP, it constructs a chronological chain showing how the attack progressed through all three detection layers: Suricata detecting network activity, Wazuh generating host-level alerts, and Fail2ban issuing the enforcement response. This query is the primary Blue Team tool for post-incident analysis following Red Team exercises.

### 6.8 Daily Security Summary Report

```spl
index=wazuh_alerts OR index=suricata OR index=smarthome
earliest=-24h latest=now
| eval tool=case(
    index="wazuh_alerts", "Wazuh",
    index="suricata",     "Suricata",
    index="smarthome" AND sourcetype="fail2ban", "Fail2ban",
    index="smarthome" AND sourcetype="ufw",      "UFW",
    true(), "Other"
  )
| stats count AS event_count by tool
| append [
    search index=wazuh_alerts earliest=-24h
    | stats count AS total_wazuh_alerts,
             max(rule.level) AS max_severity
  ]
| append [
    search index=smarthome sourcetype=fail2ban Ban earliest=-24h
    | rex field=_raw "Ban\s+(?P<banned_ip>\d+\.\d+\.\d+\.\d+)"
    | stats dc(banned_ip) AS unique_ips_banned
  ]
```

**Purpose:** A daily operational summary showing total event volumes by tool and the total number of unique IPs banned by Fail2ban in the last 24 hours. Used as the starting point for the daily Blue Team review cycle described in [[11.2_Methodology_and_Implementation#Continuous Monitoring Operating Procedure|Section 11.2.3]].
