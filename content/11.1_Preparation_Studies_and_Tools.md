# 11.1 Preparation, Studies, and Tools — Cybersecurity Layer

## Overview

The cybersecurity architecture of this project was designed around a **defence-in-depth** philosophy, wherein multiple independent security controls are layered such that the failure or bypass of any single control does not result in a total system compromise. This section documents the theoretical background, selection rationale, and preparatory configuration of every security tool deployed on the [[PC Server|Ubuntu 24.04 edge server]]. The tools operate as an integrated ecosystem rather than as isolated utilities, with log data flowing from low-level system events upward into a centralised [[Splunk]] SIEM for correlation and alerting.

The deployment context is a headless Ubuntu 24.04 LTS server acting as the central controller for a [[Smart Home Architecture|Hybrid Edge-Computing AI Smart Home]]. The threat surface encompasses the [[MQTT Broker|Mosquitto MQTT broker]] (the primary communication channel for all [[ESP32]] sensor nodes), the SSH administrative interface, and the host operating system itself. Security controls were selected and configured to address each of these attack surfaces explicitly.

---

## 11.1.1 Network and Host Architecture Context

Before detailing individual tools, it is necessary to establish the network topology against which all security controls were designed. The [[PC Server]] occupies a dual role: it is simultaneously a **data plane** node (running the [[Mosquitto]] broker and [[AI Decision Engine]]) and a **security plane** node (running the full SIEM stack). This co-location was a deliberate architectural decision driven by the single-host constraint of the graduation project scope.

The following port allocation was established prior to any tool installation and serves as the authoritative reference for all firewall and access control configurations:

| Port | Protocol | Service | Bound Interface | Access Scope |
|------|----------|---------|-----------------|--------------|
| 1883 | TCP | MQTT (plaintext) | `127.0.0.1` | Localhost only |
| 8883 | TCP | MQTT over TLS | `0.0.0.0` | LAN (`192.168.1.0/24`) |
| 2222 | TCP | SSH (hardened) | `0.0.0.0` | LAN only via UFW |
| 443 | TCP | Wazuh Dashboard | `0.0.0.0` (node) | Localhost only via UFW |
| 9200 | TCP | Wazuh Indexer | `[::ffff:127.0.0.1]` | Localhost only |
| 55000 | TCP | Wazuh Manager API | `0.0.0.0` | Localhost only via UFW |
| 1514 | TCP | Wazuh Agent data | `0.0.0.0` | Localhost only |
| 1515 | TCP | Wazuh Agent registration | `0.0.0.0` | Localhost only |
| 8000 | TCP | Splunk Web UI | `127.0.0.1` | Localhost only |
| 8088 | TCP | Splunk HEC | `127.0.0.1` | Localhost only |
| 8089 | TCP | Splunk API | `127.0.0.1` | Localhost only |
| 3389 | TCP | GNOME Remote Desktop | `0.0.0.0` | LAN only via UFW |

This port map directly informed the [[UFW]] ruleset, the [[Fail2ban]] jail configurations, and the [[Suricata]] network monitoring rules. Any deviation from this allocation constitutes an anomaly detectable by the security stack.

## 11.1.2 Wazuh — Host-Based Intrusion Detection and SIEM

### Background and Selection Rationale

[[Wazuh]] is an open-source, enterprise-grade security platform that combines **Host-Based Intrusion Detection (HIDS)**, **Security Information and Event Management (SIEM)**, **File Integrity Monitoring (FIM)**, and **active response** capabilities within a unified agent-manager architecture. It was selected over alternative HIDS solutions (such as OSSEC or Samhain) for the following reasons:

- **Native JSON alert output**, which facilitates direct integration with [[Splunk]] via the HTTP Event Collector (HEC) without requiring intermediate parsing.
- **Built-in Wazuh Indexer** (based on OpenSearch), which provides a queryable, time-series-indexed alert store accessible through the **Wazuh Dashboard** (based on OpenSearch Dashboards).
- **MITRE ATT&CK framework mapping** embedded natively in the ruleset, enabling academic alignment of detected threats to the industry-standard adversary behaviour taxonomy.
- **Active response** support, which allows the platform to automatically execute countermeasures (such as IP blocking via `firewall-drop`) upon rule triggers, creating a closed-loop detection-and-response cycle.

### Architectural Decomposition

The Wazuh deployment on the [[PC Server]] consists of four distinct components, all running on the same host in version `4.14.5-1`:

**Wazuh Indexer** (`wazuh-indexer`, port 9200)
A distributed search and analytics engine derived from OpenSearch. It receives processed alert data from the Wazuh Manager and stores it in time-indexed shards. It is bound exclusively to the loopback interface (`[::ffff:127.0.0.1]:9200`) to prevent external access to raw alert data.

**Wazuh Manager** (`wazuh-manager`, port 55000 for API, port 1514 for agent data, port 1515 for agent registration)
The central processing component. It receives raw log events forwarded by the Wazuh Agent, applies decoders to extract structured fields, evaluates those fields against the ruleset, and generates alerts. It also hosts the RESTful API consumed by the Wazuh Dashboard.

**Wazuh Agent** (`wazuh-agent`)
A lightweight daemon running on the monitored host (the [[PC Server]] itself, registered as agent `001`). It performs three primary functions: (1) tailing monitored log files and forwarding new entries to the Manager, (2) executing real-time FIM scans on designated directories using `inotify`, and (3) running periodic rootkit and system integrity checks.

**Wazuh Dashboard** (`wazuh-dashboard`, port 443)
A web-based visualisation interface exposing alert timelines, FIM events, MITRE ATT&CK heat maps, agent status, and the active response log. It communicates with the Manager exclusively through the port 55000 API endpoint using the `wazuh-wui` service account.

### Log Collection Configuration

The Wazuh Agent was configured via `/var/ossec/etc/ossec.conf` to monitor the following log sources, each chosen for its direct relevance to the smart home threat model:

```xml
<!-- SSH authentication events — brute force and unauthorised access -->
<localfile>
  <log_format>syslog</log_format>
  <location>/var/log/auth.log</location>
</localfile>

<!-- Mosquitto MQTT broker — connection, auth failure, ACL violation events -->
<localfile>
  <log_format>syslog</log_format>
  <location>/var/log/mosquitto/mosquitto.log</location>
</localfile>

<!-- Suricata IDS — network-level threat detection in JSON event format -->
<localfile>
  <log_format>json</log_format>
  <location>/var/log/suricata/eve.json</location>
</localfile>

<!-- Fail2ban — IP ban/unban events -->
<localfile>
  <log_format>syslog</log_format>
  <location>/var/log/fail2ban.log</location>
</localfile>

<!-- UFW firewall — dropped packet events -->
<localfile>
  <log_format>syslog</log_format>
  <location>/var/log/ufw.log</location>
</localfile>

<!-- General system log -->
<localfile>
  <log_format>syslog</log_format>
  <location>/var/log/syslog</location>
</localfile>
```

### File Integrity Monitoring Scope

FIM was configured with `realtime="yes"` on all directories containing security-critical configuration files. This ensures that any modification to these files generates an immediate Wazuh alert (rule SID 550 for modifications, SID 554 for additions), regardless of which process performed the change:

```xml
<syscheck>
  <frequency>300</frequency>
  <alert_new_files>yes</alert_new_files>

  <!-- MQTT broker configuration and TLS certificates -->
  <directories check_all="yes" report_changes="yes" realtime="yes">
    /etc/mosquitto
  </directories>
  <directories check_all="yes" report_changes="yes" realtime="yes">
    /etc/mosquitto/certs
  </directories>

  <!-- AI engine source code -->
  <directories check_all="yes" report_changes="yes" realtime="yes">
    /opt/smarthome
  </directories>

  <!-- Security tool configurations -->
  <directories check_all="yes" report_changes="yes">
    /etc/fail2ban
  </directories>
  <directories check_all="yes" report_changes="yes">
    /etc/suricata
  </directories>
  <directories check_all="yes" report_changes="yes">
    /etc/ssh
  </directories>
</syscheck>
```

The rationale for including `/etc/mosquitto/certs` as a realtime FIM target is specifically to detect the preparation phase of a **Man-in-the-Middle (MITM) attack**, wherein an adversary would need to replace or add a certificate before intercepting MQTT traffic. Any modification to this directory triggers custom [[Wazuh Rules|rule 100006]] at severity level 14 (Critical).

### Custom Decoder — Mosquitto Log Parsing

Wazuh ships with no built-in decoder for Mosquitto logs. A custom decoder was authored at `/var/ossec/etc/decoders/mosquitto_decoder.xml` to parse the specific log format produced by [[Mosquitto]] under the `log_timestamp_format %Y-%m-%dT%H:%M:%S` configuration. The decoder hierarchy consists of a root decoder that anchors on the ISO 8601 timestamp prefix, and seven child decoders that extract structured fields from specific event types including authentication failures, ACL violations, TLS errors, and abnormal disconnections. This decoder is the prerequisite for all custom MQTT security rules.

### Custom Rule Set — Smart Home Threat Model

A custom rule file was authored at `/var/ossec/etc/rules/smarthome_rules.xml` covering rule IDs `100001` through `100010`. The following table summarises the rule set with its MITRE ATT&CK mappings:

| Rule ID | Level | Description | MITRE ATT&CK |
|---------|-------|-------------|--------------|
| 100001 | 5 | MQTT single authentication failure | T1110 — Brute Force |
| 100002 | 10 | MQTT brute force: 5+ failures in 120 s | T1110.001 — Password Guessing |
| 100003 | 8 | MQTT ACL violation — unauthorised topic access | T1078 — Valid Accounts |
| 100004 | 6 | MQTT anonymous connection attempt | T1133 — External Remote Services |
| 100005 | 12 | Mosquitto configuration file modified | T1565.001 — Stored Data Manipulation |
| 100006 | 14 | MQTT TLS certificate modified | T1557 — Adversary-in-the-Middle |
| 100007 | 6 | TLS handshake error — cert mismatch or scanning | T1040 — Network Sniffing |
| 100008 | 9 | Multiple TLS errors — certificate scanning | T1595.002 — Active Scanning |
| 100009 | 5 | Repeated abnormal disconnections | T1499 — Endpoint DoS |
| 100010 | 10 | Mosquitto broker unexpected restart | T1489 — Service Stop |

Active response was configured to invoke the `firewall-drop` command automatically upon triggering rules 100002 (MQTT brute force) and 5712 (built-in SSH brute force rule), with ban durations of 3,600 and 7,200 seconds respectively.

## 11.1.3 Suricata — Network-Based Intrusion Detection

### Background and Selection Rationale

[[Suricata]] is a high-performance, open-source **Network Intrusion Detection and Prevention System (NIDS/NIPS)** maintained by the Open Information Security Foundation (OISF). It operates by performing deep packet inspection (DPI) on network traffic captured from a specified interface, matching packet content and flow metadata against a structured rule set, and emitting alerts in a machine-readable JSON format (`eve.json`).

Suricata was selected as the network-layer complement to Wazuh's host-layer detection for the following reasons:

- **MQTT protocol awareness**: Suricata 7.x includes a built-in MQTT application-layer parser, enabling rule matching on MQTT-specific fields (client ID, topic, payload, QoS level) rather than only TCP/IP metadata.
- **EVE JSON output**: The `eve.json` unified event log is directly consumable by both Wazuh (via `<localfile>`) and [[Splunk]] (via the universal forwarder or HEC), eliminating the need for log transformation middleware.
- **Community ruleset (ET/Open)**: The Emerging Threats Open ruleset provides thousands of pre-authored signatures for common attack patterns, updated daily, and installable via `suricata-update`.
- **AF_PACKET capture mode**: Enables zero-copy packet capture on Linux, minimising the performance overhead of continuous traffic inspection on the edge server.

### Configuration

Suricata was installed from the official OISF stable PPA (`ppa:oisf/suricata-stable`) to ensure access to version 7.x with full MQTT parser support. The primary configuration file at `/etc/suricata/suricata.yaml` was modified as follows:

**Network scope definition:**
```yaml
vars:
  address-groups:
    HOME_NET: "[192.168.1.0/24]"
    EXTERNAL_NET: "!$HOME_NET"
```

**Capture interface** (set to the active LAN interface detected via `ip route`):
```yaml
af-packet:
  - interface: ens33   # substituted with actual interface at deployment
    cluster-id: 99
    cluster-type: cluster_flow
    defrag: yes
```

**EVE JSON output with MQTT event logging:**
```yaml
outputs:
  - eve-log:
      enabled: yes
      filetype: regular
      filename: /var/log/suricata/eve.json
      community-id: true
      types:
        - alert
        - http
        - dns
        - tls
        - flow
        - ssh
        - mqtt
```

The `community-id: true` field generates a deterministic hash of the 5-tuple (src IP, dst IP, src port, dst port, protocol) for each flow. This hash appears identically in Suricata's `eve.json`, Wazuh's alert output, and Splunk's indexed events, enabling **cross-tool event correlation** by joining on this field.

### Custom Rules

A project-specific rule file was authored at `/etc/suricata/rules/smarthome.rules` addressing threats specific to the smart home architecture. Key rules include detection of MQTT connection flooding, unencrypted MQTT traffic originating from outside localhost (indicating a misconfigured or rogue device bypassing the TLS listener), oversized MQTT payloads suggestive of data exfiltration, and port scanning activity targeting port 8883.

## 11.1.4 Splunk Enterprise — Security Information and Event Management

### Background and Selection Rationale

[[Splunk]] Enterprise is an industry-standard platform for ingesting, indexing, searching, and visualising machine-generated data at scale. In this architecture it serves as the **top-level aggregation and correlation layer**, receiving data from all other security tools and providing a unified query interface across the entire event corpus.

Splunk was selected over open-source alternatives (such as the ELK stack) for its **Search Processing Language (SPL)**, which enables complex statistical correlations across heterogeneous data sources in a single query without requiring pre-defined schemas. The free tier (up to 500 MB/day of indexing) is sufficient for the data volumes generated by a residential smart home deployment.

### Installation and Service Configuration

Splunk Enterprise 9.2.1 was installed from the official `.deb` package to `/opt/splunk/`. The `Splunkd` systemd service was enabled for boot-start. The web interface was bound exclusively to `127.0.0.1:8000` via `/opt/splunk/etc/system/local/web.conf` to prevent LAN exposure of the administrative interface.

### Index Architecture

Four purpose-specific indexes were created to partition data by source domain, enabling efficient targeted searches and independent retention policies:

| Index | Contents | Max Size |
|-------|----------|----------|
| `mqtt_logs` | Mosquitto broker log events | 2 GB |
| `wazuh_alerts` | Wazuh Manager alert JSON | 5 GB |
| `suricata` | Suricata EVE JSON events | 2 GB |
| `smarthome` | Auth, UFW, Fail2ban, Syslog | 5 GB |

### Data Inputs

Universal monitor inputs were configured for all log sources. JSON-format logs (Wazuh alerts, Suricata EVE) were assigned `KV_MODE = json` in `props.conf` to enable automatic field extraction without requiring explicit field transforms. The `TIME_FORMAT` and `TIME_PREFIX` directives were set per sourcetype to ensure correct event timestamping from the original log entry rather than the Splunk ingestion time.

### Wazuh Integration via HTTP Event Collector

To achieve real-time alert forwarding from [[Wazuh]] to [[Splunk]], the Splunk **HTTP Event Collector (HEC)** was enabled on port 8088. A dedicated HEC token was generated for the `wazuh_alerts` index. The Wazuh Manager's `ossec.conf` was extended with a `<integration>` block specifying the HEC endpoint and token, configured to forward all alerts at severity level 3 and above in JSON format. This integration creates a live pipeline wherein a Wazuh rule trigger appears in Splunk within seconds of the underlying log event.

## 11.1.5 Mosquitto MQTT Broker — Secure Configuration

### Background

[[Mosquitto]] is the reference implementation of the MQTT protocol broker, maintained by the Eclipse Foundation. MQTT (Message Queuing Telemetry Transport) is a lightweight publish-subscribe messaging protocol designed for constrained IoT devices operating over low-bandwidth, high-latency networks. It is the sole communication medium between the [[ESP32]] sensor nodes and the [[PC Server]].

The security posture of the Mosquitto broker is the most operationally critical element of the entire cybersecurity architecture, as the broker is the only service that accepts inbound network connections from external devices (the ESP32 nodes). A compromised broker implies full visibility into all sensor telemetry and the ability to inject false actuation commands.

### Two-Listener Architecture

The broker was configured with two distinct listeners serving different trust levels:

**Port 1883 — Plaintext listener (localhost only)**
Bound to `127.0.0.1`, this listener is inaccessible from any network interface. It exists exclusively for local processes on the [[PC Server]] — specifically the [[AI Decision Engine]] (`ai_engine.py`) and the [[ThingsBoard Gateway]] (`tb-gateway`) — to communicate with the broker without the overhead of TLS, since these are inter-process communications on a single host.

**Port 8883 — TLS listener (LAN-facing)**
Bound to all interfaces (`0.0.0.0`) and restricted to the LAN subnet by [[UFW]]. All [[ESP32]] devices connect exclusively through this listener. The TLS configuration enforces TLS 1.3 (`tls_version tlsv1.3`), rejecting all older protocol versions.

### TLS Certificate Infrastructure

A three-tier PKI was established using OpenSSL:

1. **Certificate Authority (CA)**: A self-signed root CA (`ca.crt`) with a 10-year validity, generated with a 4096-bit RSA key. This CA certificate is the trust anchor distributed to all ESP32 devices as a hardcoded constant in their firmware.
2. **Broker Certificate**: Signed by the project CA, with a Subject Alternative Name (SAN) extension including the server's LAN IP address. SAN is required by modern TLS client implementations; a certificate without it will be rejected by current ESP32 Arduino TLS libraries.
3. **Private Key**: A 2048-bit RSA key stored at `/etc/mosquitto/certs/broker.key` with permissions `0600`, readable only by the `mosquitto` system user.

### Authentication and Access Control

Anonymous access was disabled globally (`allow_anonymous false`). A dedicated credential was provisioned for each client identity using `mosquitto_passwd`, with passwords stored as PBKDF2-SHA512 hashes in `/etc/mosquitto/passwd`. The following identities were created: `esp32_sensor_1` through `esp32_sensor_4`, `ai_engine`, and `tb_gateway`.

Topic-level access control was implemented via `/etc/mosquitto/acl` using the principle of least privilege: each ESP32 identity is restricted to write access on its own sensor topic subtree (`home/sensors/esp32_N/#`) and read access on its own actuator subtree (`home/actuators/esp32_N/#`). The `ai_engine` identity holds read access on all sensor topics and write access on all actuator topics. The `tb_gateway` identity holds read-only access on all topics for cloud forwarding. Any attempt by a client to publish or subscribe to a topic outside its ACL entry generates a Mosquitto log entry parsed by the [[Wazuh Rules|custom Wazuh decoder]] and triggers rule 100003.

### SIEM-Oriented Logging Configuration

The Mosquitto logging configuration was specifically engineered for Wazuh compatibility. All relevant log type categories were enabled (`error`, `warning`, `notice`, `information`, `subscribe`, `unsubscribe`, `connect`, `disconnect`) and directed to a dedicated file at `/var/log/mosquitto/mosquitto.log`. The `log_timestamp_format` was set to `%Y-%m-%dT%H:%M:%S` (ISO 8601), which is the format expected by Wazuh's timestamp parser. The `connection_messages true` directive ensures that every connection and disconnection — including failed authentication attempts — is explicitly recorded, making the log file suitable for brute-force detection.

## 11.1.6 Fail2ban — Automated Threat Response

### Background and Selection Rationale

[[Fail2ban]] is a log-parsing intrusion prevention framework that monitors designated log files for patterns indicative of attack behaviour and automatically applies firewall bans to source IP addresses exceeding configurable failure thresholds. It operates as the **automated first-response layer** of the defence stack, translating detected attack patterns into immediate network-level countermeasures without requiring human intervention.

In this architecture, Fail2ban serves as the bridge between the **detection layer** (Mosquitto and SSH logs) and the **enforcement layer** ([[UFW]] firewall), providing sub-minute response times to credential brute-force attacks.

### Jail Configuration

Fail2ban was configured with four jails, each targeting a distinct attack surface:

**`sshd` jail** — Monitors `/var/log/auth.log` for SSH authentication failures using the built-in `sshd` filter. Configured with `maxretry = 3` and `bantime = 7200` (2 hours). The SSH port was updated to `2222` in the jail configuration to reflect the hardened SSH port assignment.

**`mosquitto` jail** — Monitors `/var/log/mosquitto/mosquitto.log` using a custom filter at `/etc/fail2ban/filter.d/mosquitto.conf`. The filter regex targets the `"failed to connect, not authorised"` log pattern emitted by Mosquitto on authentication failure. Configured with `maxretry = 3`, `findtime = 300` (5-minute window), and `bantime = 3600` (1 hour).

**`wazuh-api` jail** — Monitors `/var/ossec/logs/api.log` for repeated failed API authentication attempts against the Wazuh Manager API on port 55000, preventing credential attacks against the SIEM management interface itself.

**`suricata` jail** — Monitors `/var/log/suricata/fast.log` for Suricata alert entries. Configured with `maxretry = 1`, meaning a single Suricata alert triggers an immediate ban. This provides a direct integration between network-level threat detection and firewall enforcement.

All jails use `banaction = ufw`, which invokes UFW's deny rule mechanism rather than direct iptables manipulation, maintaining consistency with the project's firewall management approach.

## 11.1.7 UFW — Stateful Firewall and Network Perimeter

### Background

[[UFW]] (Uncomplicated Firewall) is the standard frontend for `iptables`/`nftables` on Ubuntu systems. It provides a simplified interface for managing stateful packet filtering rules. In this architecture, UFW functions as the **network perimeter enforcement layer**, implementing the port access policy defined in [[#11.1.1 Network and Host Architecture Context]].

### Policy

The firewall was configured with a **default-deny** posture: all inbound connections are rejected unless explicitly permitted by a named rule. Outbound connections are permitted by default to allow the server to initiate connections to [[ThingsBoard]] cloud and to retrieve security updates. Forward policy is set to deny.

The complete applied ruleset restricts all Wazuh-internal ports (9200, 55000, 1514, 1515, 443), Splunk ports (8000, 8088, 8089), and the plaintext MQTT port (1883) to `127.0.0.1` source addresses exclusively. The TLS MQTT port (8883) is restricted to the LAN subnet (`192.168.1.0/24`) to accept connections from ESP32 devices while rejecting any attempt from outside the local network. UFW logging was enabled at `medium` verbosity, directing firewall drop events to `/var/log/ufw.log` where they are monitored by both [[Wazuh]] and [[Splunk]].

## 11.1.8 SSH Hardening

### Rationale

The SSH service is the sole interactive administrative interface to the [[PC Server]]. As such, it represents the highest-value target for an adversary seeking privileged access to the host. A successful SSH compromise would allow an attacker to disable all security controls, exfiltrate all sensor data, and inject malicious actuation commands. The default Ubuntu SSH configuration was replaced with a hardened configuration addressing each of these risks.

### Applied Hardening Measures

The following modifications were made to `/etc/ssh/sshd_config`:

| Directive | Value | Security Purpose |
|-----------|-------|-----------------|
| `Port` | `2222` | Moves service off the default port, reducing automated scan noise |
| `PermitRootLogin` | `no` | Prevents direct root access; forces use of `sudo` |
| `PasswordAuthentication` | `no` | Disables password-based authentication entirely; requires Ed25519 key |
| `PubkeyAuthentication` | `yes` | Enables public key authentication as the sole permitted method |
| `MaxAuthTries` | `3` | Limits authentication attempts per connection before disconnection |
| `MaxSessions` | `3` | Caps concurrent sessions per connection |
| `LoginGraceTime` | `30` | Disconnects unauthenticated sessions after 30 seconds |
| `X11Forwarding` | `no` | Disables X11 forwarding, closing a lateral movement vector |
| `AllowTcpForwarding` | `no` | Prevents use of SSH as a tunnelling proxy |
| `AllowAgentForwarding` | `no` | Prevents credential forwarding attacks |
| `Protocol` | `2` | Enforces SSH protocol version 2; version 1 is cryptographically broken |
| `Banner` | `/etc/ssh/ssh_banner` | Displays legal warning banner before authentication |
| `UseDNS` | `no` | Eliminates reverse DNS lookup delay at connection time |

A legal warning banner was deployed at `/etc/ssh/ssh_banner` and an Ed25519 key pair was generated as the sole permitted authentication credential. The [[UFW]] ruleset was simultaneously updated to restrict SSH access to port 2222, and the [[Fail2ban]] `sshd` jail was reconfigured to monitor the new port.

## 11.1.9 Kernel-Level Hardening via sysctl

Beyond application-layer security controls, a set of kernel network stack parameters was applied via `/etc/sysctl.d/99-smarthome-hardening.conf` to harden the OS against network-layer attacks. Key parameters include:

- `net.ipv4.tcp_syncookies = 1` — enables SYN cookie protection against SYN flood DoS attacks.
- `net.ipv4.conf.all.rp_filter = 1` — enables reverse path filtering to reject IP-spoofed packets.
- `net.ipv4.conf.all.accept_redirects = 0` and `net.ipv4.conf.all.send_redirects = 0` — disables ICMP redirect processing, preventing route table manipulation.
- `net.ipv4.conf.all.log_martians = 1` — logs packets with impossible source addresses to `/var/log/syslog`, providing visibility into spoofing attempts.
- `net.ipv4.icmp_echo_ignore_broadcasts = 1` — prevents the host from responding to broadcast ping requests, reducing its utility as a Smurf attack amplifier.

## 11.1.10 Auditd — Kernel-Level Audit Trail

[[Auditd]] was installed and configured to provide a kernel-level audit trail for file access events on security-critical paths, operating independently of and complementary to Wazuh's FIM. While Wazuh's FIM detects that a file was changed, `auditd` records the system call, the process name, the user ID, and the timestamp of the access attempt — information that is not available at the inotify level. Audit rules were written to `/etc/audit/rules.d/smarthome.rules` targeting `/etc/mosquitto`, `/opt/smarthome`, `/etc/ssh/sshd_config`, and `/var/ossec/etc/ossec.conf`.
